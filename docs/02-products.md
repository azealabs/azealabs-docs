# AZea Labs - Products

## 1. Product Overview

AZea Labs is an ecosystem composed of multiple connected products.

Each product has a specific purpose, audience, and growth path.

The initial products are:

* AZea Labs Portfolio
* QA Labs
* QA Academy
* Future Projects

The goal is to keep each product separated by responsibility, while keeping all products connected under the AZea Labs brand.

## 3. QA Labs

### Purpose

QA Labs is a practice platform for QA Automation.

It will provide demo web pages, applications, APIs, and testing scenarios designed specifically for automation practice.

QA Labs is not the academy. It is the practice environment that can be used by students, professionals, and Antonio Zea's future courses.

### Primary Audience

* QA Automation Engineers
* SDET professionals
* Automation students
* Developers learning testing
* Future QA Academy students

### Main Goals

* Provide stable demo pages for automation practice.
* Provide realistic testing challenges.
* Support Selenium, Cypress, Playwright, Robot Framework, and other tools.
* Provide API testing practice.
* Provide controlled scenarios that do not depend on external demo websites.
* Serve as the hands-on platform for future QA Automation classes.

### Initial Features

* Login demo
* Registration demo
* Forms demo
* Tables demo
* Upload demo
* Download demo
* Dashboard demo
* API demo
* Basic documentation for each demo page

### Future Features

* Advanced login scenarios
* Role-based access scenarios
* Shopping flow
* Checkout flow
* Shadow DOM practice
* iframe practice
* Drag and drop practice
* Dynamic elements
* Infinite scroll
* WebSocket scenarios
* GraphQL practice
* Test data reset
* Challenge mode
* Automation examples by framework

### Possible Subdomain


## 4. QA Academy

### Purpose

QA Academy is a future educational platform focused on teaching QA Automation.

It will use QA Labs as the practice platform.

The academy is not intended to teach programming from zero. Its main focus will be automation, testing strategies, tooling, frameworks, and real-world QA workflows.

### Primary Audience

* People who already know basic programming and want to learn automation.
* Manual testers moving into automation.
* Junior QA Automation Engineers.
* QA professionals who want to practice with real scenarios.

### Main Goals

* Teach QA Automation using practical exercises.
* Use QA Labs as the hands-on environment.
* Provide structured learning paths.
* Generate an additional income stream.
* Build authority in the QA Automation space.

### Initial Possible Courses

* Selenium WebDriver Fundamentals
* Selenium with Java or C#
* Selenium with Python
* Cypress Fundamentals
* Playwright Fundamentals
* Robot Framework for Automation
* API Testing Fundamentals
* Automation Framework Design
* CI/CD for QA Automation

### Future Features

* Course platform
* Student dashboard
* Downloadable material
* Certificates
* Workshops
* Paid courses
* Private community
* Mentoring sessions

### Possible Subdomain

```text
academy.azealabs.com
```

## 6. Product Separation Strategy

Each product should have a clear responsibility.

### Portfolio

Professional identity and central hub.

### QA Labs

Practice platform for automation.

### QA Academy

Educational business and teaching platform.

### Future Projects

Learning, experimentation, and portfolio growth.

This separation helps avoid confusion and keeps the architecture scalable.

---

## 7. Product Connection Strategy

Even though the products are separated, they should be connected through:

* Shared brand identity.
* Shared visual language.
* Shared navigation where appropriate.
* Links between platforms.
* Consistent documentation style.
* GitHub organization.
* Future shared design system.

Example structure:

```text
azealabs.com              -> Main hub and portfolio
labs.azealabs.com         -> QA Labs practice platform
academy.azealabs.com      -> Future QA Academy
docs.azealabs.com         -> Future documentation
api.azealabs.com          -> Future API services
```

---

## 8. First Product Priority

The first product to build is:

```text
AZea Labs Portfolio
```

Reason:

* It establishes the brand.
* It provides a public professional presence.
* It links all future products.
* It is simpler than QA Labs.
* It allows early deployment of `azealabs.com`.

The second product to build is:

```text
QA Labs MVP
```

Reason:

* It is the main technical product.
* It supports automation practice.
* It will later support QA Academy.
* It provides strong portfolio value.

QA Academy should come after QA Labs has enough practice scenarios.
